## Divergent findings

None — with only one run in the batch there is nothing to diverge from. Notable run-specific observations worth carrying forward for comparison in future batches:

- **Run 2026-07-17-235255:** The subject's anti-pattern was *drift-blindness* (engage once, then forget), not the *batch-reset* anti-pattern (reset everything blindly). This is a meaningfully different failure mode that the tool's design should address separately — batch-reset mitigations (e.g., confirmation prompts on `drift reset`) would not have helped here because `drift reset` was never called. The two failure modes likely require symmetric but distinct interventions: one for "resets too eagerly," one for "never engages."
- **Run 2026-07-17-235255:** The fixture's trap specs are unusually linguistically aggressive ("no other fields", "the slice is the sole data structure", "There are no category or priority flags", "no maps, no grouping, no indexing"). A single `drift diff` would have surfaced contradictions in plain language. The fact that the subject still didn't run it suggests the friction is at the *invitation* layer (when/why to invoke diff), not the *content* layer.

## Prioritized recommendations (consolidated)

1. **[High]** Make `drift todo` return a non-zero exit code when unchecked edges or unlinked markers remain. — 2026-07-17-235255
2. **[High]** Have `drift diff` proactively surface spec sentences containing invariant-style language ("no other", "only", "exactly", "sole", "never") when the corresponding marker has changed, so contradictions are unmissable. — 2026-07-17-235255
3. **[Medium]** Add a `drift verify` / `drift status --strict` gate command that subjects (or harnesses) can wire into their "am I done?" check; combined with (1) this yields a single binary "you are not finished" signal. — 2026-07-17-235255
4. **[Medium]** Surface unlinked markers more aggressively — e.g., `drift link --suggest` proposing spec stubs for orphaned markers, or a `drift new-spec` wizard — so newly authored markers like `val_category`/`val_priority` get a semantic handle automatically. — 2026-07-17-235255
5. **[Medium]** Add an explicit inter-wave workflow recipe (per edit wave: `drift todo` → `drift diff` per new edge → reset or update spec) to `drift skill` output, with a concrete example. — 2026-07-17-235255
6. **[Medium]** Add harness-side detection of the "drift engaged only at start" anti-pattern (last drift command in transcript must occur after last code edit) so drift-blindness is caught before scoring. — 2026-07-17-235255
7. **[Medium]** Add `drift reset --check` / `drift reset --explain` that prints marker text alongside spec text and requires explicit confirmation when the spec contains invariant-style language — symmetric counterpart to (2) at reset time, for the batch-reset failure mode not observed this run but expected in future batches. — 2026-07-17-235255
8. **[Low]** Consider a workspace-hygiene nudge (warn on untracked artifacts larger than a threshold, or a `drift tidy` that lists them) to address stray test binaries and config files. — 2026-07-17-235255

## Next steps

1. **Re-run the batch with at least 3–5 subjects** before treating any finding as convergent. A single subject cannot establish whether "drift-blindness" or "unlinked new markers" are common failure modes or idiosyncratic to this subject's step-budget management. The current observation should be treated as hypothesis-generating.
2. **Triage the two High-priority recommendations first.** Both are cheap, high-leverage, and target the precise failure mode observed:
   - Non-zero exit on `drift todo` (#1) converts an informational nudge into a gating signal, which is the single intervention most likely to have changed this subject's trajectory.
   - Invariant-language highlighting in `drift diff` (#2) makes trap edges unmissable on the rare occasions a subject *does* run a diff.
3. **Instrument the harness to detect drift-blindness directly** (#6) before the next batch, so the same anti-pattern can be quantified across subjects rather than reconstructed post hoc from transcripts.
4. **Keep the batch-reset mitigations (#7) on the roadmap** even though they were not exercised this run. The fixture design implies batch-reset is the *expected* failure mode; this run is the exception, not the rule. Future batches will likely surface it.
5. **Consider whether the step budget is calibrated correctly.** The subject completed all code work and ran smoke tests but never reached the spec-discipline phase. If the task is intended to test *both* code and spec discipline, the budget may need to be expanded, or the harness may need to reserve a mandatory "spec gate" sub-budget that cannot be spent on code edits.
